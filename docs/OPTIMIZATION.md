# 追補タスク：軽量化・最適化（第1段＋第2段）(OPTIMIZATION.md)

GitHub 公開を見据えた最適化。**リスクの低い2段階のみ**。`wm-core` の移植性（no_std・固定長 Grid）には手を付けない。前回同様このファイルを読んで実装・検証してほしい。設計の鉄則（wm-core は無改変で RISC-V に載る／時刻は外部から u64 で渡す）は厳守。

前提：全機能（地図・タイムライン・ラベル・トグル・詳細ズーム）は実装済み・動作確認済み。本タスクは挙動を変えずに軽く・速くするだけ。**機能の見た目・操作は一切変えない**こと。

---

## 第1段：Cargo リリースプロファイル（バイナリ縮小・リスクほぼ0）

ワークスペースルートの `Cargo.toml`（`[workspace]` がある方）に以下を追加。コードは一切触らない。

```toml
[profile.release]
opt-level = "z"       # サイズ優先。速度が気になるなら "s" に変えてよい
lto = true            # リンク時最適化（重複コード除去）
codegen-units = 1     # 最適化を最大化（ビルドは遅くなるが実行は速い/小さい）
strip = true          # デバッグシンボル除去
```

**重要：`panic = "abort"` は入れないこと。** このアプリは panic hook で端末（raw mode / 代替画面）を復帰させている。abort にすると hook が走らずに即終了し、端末が壊れた状態で残る経路ができる。サイズより端末復帰の安全を優先する。

### 確認
- `cargo build -p wm-tui --release` が通る。
- ビルド前後で `ls -la target/release/amescii` のサイズを比較し、縮小していることを記録（README か lessons.md に「最適化前 X MB → 後 Y MB」）。
- 実起動して全機能が変わらず動く（地図・雨雲・ラベル・t トグル・ズーム・q 復帰）。
- RISC-V embedded ビルドが引き続き緑（プロファイルは wm-core のロジックに影響しないが念のため確認）。

> 注：`wm-esp32` 側は別プロファイル（`opt-level="s"`, `lto=true` が既にあるはず）。今回のワークスペース release プロファイルは PC バイナリ向け。両者が衝突しないこと。

## 第2段：タイルキャッシュ（再取得削減・体感速度＆通信量・リスク低）

### 現状の無駄
パン/ズームのたびに、同じ (z,x,y) タイルを何度も取得している。特にパンで行ったり来たりすると、直前に取ったタイルを再取得している。地図タイル（地理院ベクトル）は内容がほぼ不変なのでキャッシュが非常に効く。

### キャッシュ方針（データ種別で分ける・重要）
- **地図ベクトルタイル（experimental_bvmap, .pbf）**：内容がほぼ不変。**メモリ + 任意でディスク**にキャッシュしてよい。再起動後も使える。
- **雨雲タイル（JMA ナウキャスト PNG）**：5 分で更新され、basetime/validtime がキーに含まれる。**メモリのみ（セッション内）**。ディスクに残すと古い雨雲を掴むリスクがあるため**ディスク永続化しない**。
  - ただし雨雲タイル URL は basetime/validtime を含むので、メモリキャッシュのキーにそれらを含めれば「同一フレームの再取得」を防げる（タイムライン再生でコマを行き来しても再取得しない）。これは安全。

### 実装場所
`wm-sources` に薄いキャッシュ層を足す。`wm-core` は触らない。

#### メモリキャッシュ（最小実装）
```rust
// wm-sources/src/cache.rs（新規）
//! タイルのメモリキャッシュ。上限付き LRU。HTTP 取得の前段。

use std::collections::HashMap;

/// バイト列を返すタイル取得の薄いキャッシュ。
/// キーは完全な URL（z/x/y や basetime/validtime を含むので衝突しない）。
pub struct TileCache {
    map: HashMap<String, Vec<u8>>,
    order: Vec<String>,   // LRU 用（簡易）。VecDeque でも可
    capacity: usize,      // 最大エントリ数（例：地図 256, 雨雲 256）
}

impl TileCache {
    pub fn new(capacity: usize) -> Self { /* … */ }
    pub fn get(&mut self, url: &str) -> Option<&Vec<u8>>;
    pub fn put(&mut self, url: String, bytes: Vec<u8>);
    // 上限超過時は LRU で最古を捨てる
}
```

- 地図用と雨雲用で別インスタンスを持つ（容量・寿命の性質が違うため）。
- 取得経路を「キャッシュ get →無ければ HTTP→ put」に変える。`basemap.rs` の `fetch_lines`/`fetch_labels_ja` と `radar.rs` のタイル取得（`fetch_frame_grid`）が対象。
- スレッド共有が必要なら `Arc<Mutex<TileCache>>` で包む（取得は別 task で走るため）。Mutex のロック時間はタイル 1 枚の get/put だけにして短く保つ。

#### ディスクキャッシュ（地図のみ・任意）
やるなら地図ベクトルタイルだけ。`directories` crate でキャッシュディレクトリ（例：`~/.cache/amescii/tiles/bvmap/{z}/{x}/{y}.pbf`）に保存。
- get：ファイルがあれば読む。無ければ HTTP→保存。
- **雨雲はディスクに保存しない**（古いフレームを掴むため）。
- ディスクキャッシュは「あれば使う、無ければ取る」だけ。無効化（古いタイルの破棄）は、experimental_bvmap が滅多に変わらないので**当面は実装しなくてよい**（必要なら更新日時で TTL を後付け）。ただし「実験提供でデータが変わりうる」ため、キャッシュを消す手段（`~/.cache/amescii` を消すだけ）を README に一文書いておく。
- ディスクキャッシュが面倒・リスクと感じるなら**メモリキャッシュだけで十分**。優先度はメモリ＞ディスク。

### 取得の重複排除（ズーム連打対策・軽い追加）
ズームやパンを連打すると中間状態のタイルまで取りに行く。最後の状態だけ取れば無駄が減る：
- 入力（パン/ズーム）から実際の取得開始までに短いデバウンス（例 150–250ms）を入れ、連打中は取得を開始しない。最後の操作が落ち着いてから取得する。
- 実装：操作のたびに「最後の操作時刻」を更新し、取得 task は「最後の操作から N ms 経過してから」発火。または既存の取得 task をキャンセルして最新で貼り直す（`tokio` の `JoinHandle::abort`）。
- これは体感に効く（連打しても最後の 1 回しか取得しない）。やりすぎると反応が鈍く感じるので 150–250ms 程度。

### キャッシュの効果確認
- パンで往復したとき、2 回目は HTTP リクエストが飛ばない（ログ or カウンタで確認）。
- タイムライン再生でコマを往復しても、各フレームのタイルは 1 回だけ取得（メモリキャッシュ命中）。
- 体感：一度見た範囲に戻る操作が即座になる。

## 受け入れ基準
- [ ] 第1段：`cargo build -p wm-tui --release` 緑、バイナリサイズが縮小（数値を記録）。`panic="abort"` は**入っていない**。
- [ ] 全機能が最適化前と同じく動作（地図・雨雲・ラベル・t・ズーム z3-16・タイムライン再生・q 復帰）。見た目・操作の変化なし。
- [ ] 第2段：パン往復で 2 回目の取得が発生しない（キャッシュ命中）。タイムライン往復で再取得が起きない。
- [ ] 地図タイルはキャッシュ、**雨雲タイルはディスク永続化していない**（メモリのみ）。
- [ ] ズーム/パン連打で取得が最後の 1 回に集約される（デバウンス）。
- [ ] `cargo test -p wm-core` / `-p wm-sources` 緑。RISC-V embedded 緑。警告 0。
- [ ] パニック無し・端末復帰。

## 実装順序の提案
1. 第1段（Cargo プロファイル）。ビルド・起動・サイズ計測。ここで一度コミット（差分が明確）。
2. 第2段メモリキャッシュ（`cache.rs`）を地図・雨雲に適用。往復で再取得が消えることを確認。
3. デバウンス（連打対策）を追加。
4. （任意）地図のみディスクキャッシュ。リスク/手間と相談、メモリだけで十分なら省略可。
5. 受け入れ基準を確認。

各段階でコンパイル・動作確認してから次へ。**機能を変えないこと**が最優先（公開前の最適化なので、挙動が変わって新たなバグを生むのが一番避けたい）。

## やらないこと（今回のスコープ外・明示）
- `wm-core` の `Grid` 固定長（256KB×N）の削減＝**やらない**。移植性の鉄則に関わるため、公開後に別途慎重に検討する。
- グリッド解像度・コマ数の変更＝**やらない**（見た目が変わる）。
- アルゴリズム・データ源の変更＝**やらない**。

純粋に「同じ挙動を、より小さいバイナリで、より少ない再取得で」だけを達成する。
