# 追補タスク：詳細ズーム + 市町名ラベル + 雨雲オーバーズーム (DEEP_ZOOM.md)

既存 weathermap に「市・町・街区レベルまでの拡大」を入れる。前回同様このファイルを読んで実装・検証してほしい。設計の鉄則（wm-core は no_std・無改変で RISC-V に載る／時刻は外部から u64 で渡す／データの作り方は外・描画は中）は厳守。

前提：機能1（地図ベース層 lines.rs/basemap.rs）、機能2（タイムライン）、機能A（内蔵英語都市ラベル）、機能B（t トグル）は実装済み。本タスクはその上に乗せる。

## 背景：なぜ今 z10 で止まっているか
- アプリの zoom 上限が 10（`zoom_in()` が `3..=10`）。
- 理由：**JMA ナウキャストのタイルが z10 までしか配信されていない**ため、それ以上ズームすると雨雲タイルが 404 になる。雨雲に合わせて地図ズームも 10 に縛られていた。
- 一方、**地理院ベクトルタイル（experimental_bvmap）は z16 まで**配信されている。

→ **雨雲のズームと地図のズームを分離**し、地図は z16 まで精細化、雨雲は z10 を上限にオーバーズーム表示する。

## 確定仕様
1. アプリの zoom 上限を **16** に上げる（`zoom_in()` の上限を 10→16）。下限は 3 のまま。
2. 地図ベース層（lines）と都市/地名ラベルは、その zoom のベクトルタイルを取得して精細化。
3. 雨雲は **z10 を上限**とし、z11–16 では **z10 のタイルをオーバーズーム**（引き伸ばし）して表示。
4. 拡大時（高ズーム）は地理院 `label` レイヤの **日本語地名（annoChar）** を出す。広域は内蔵英語都市のまま（表記は混在するが用途が違うので可）。

## 変更1：ズーム上限の分離（app.rs / radar 取得）

### app.rs
```rust
pub fn zoom_in(&mut self)  { if self.zoom < 16 { self.zoom += 1; } }  // 10→16
pub fn zoom_out(&mut self) { if self.zoom > 3  { self.zoom -= 1; } }
```

### 雨雲タイルのズームクランプ（radar.rs）
雨雲取得で使うズームを **min(app_zoom, 10)** にクランプする定数を入れる：
```rust
pub const RADAR_MAX_ZOOM: u8 = 10;
```
- `fetch_radar_timeline` / `fetch_frame_grid` がタイル URL を組む際、`z = app_zoom.min(RADAR_MAX_ZOOM)` を使う。
- ただし **Grid が覆う地理範囲（bbox）は app の現在 bbox のまま**。つまり「z10 のタイルから、現在の（もっと狭い）bbox に対応する部分を拡大して読む」ことになる。
  - 実装：既存の「ピクセル→グリッドセル写像」で、タイル取得は z10、bbox 正規化は現在 bbox。z10 のタイルピクセルのうち、現在 bbox に入るピクセルだけを Grid に書く。z11+ では現在 bbox が z10 タイルのごく一部になるので、少ないピクセルが Grid に引き伸ばされて入る（＝オーバーズーム、粗いが雨は見える）。
  - **重要**：geo.rs の投影は「実際に取得したズーム(z10)」で計算する箇所と「表示の現在ズーム」で計算する箇所を取り違えないこと。タイル座標・タイル内ピクセル→緯度経度は z10 で、緯度経度→グリッドセルは現在 bbox で行う。両者は緯度経度を介してつながる。

> これにより z11–16 でも雨雲が（粗く）出る。雨雲を消したい場合は既存の `t` トグルで OFF にできる。

### 地図・ラベルのズーム
- lines（地図）と label（地名）のタイル取得は **現在 app_zoom そのまま**（最大 16）。ただし bvmap は z16 まで、`coastline` が無いズーム帯のフォールバック（waterarea）は既存実装どおり。z14–16 は coastline が再び入るので素直に出る。

## 変更2：拡大時の日本語地名（annoChar）

### 方針
高ズーム（例 z>=11）では地理院 `label` レイヤから日本語地名を取得して描画。広域（z<11）は従来の内蔵英語都市（places.rs）。

### label レイヤの実属性（前回判明・要再確認）
前回の調査で、`experimental_bvmap` の `label` レイヤは **`name`（ローマ字）を持たず**、日本語系の属性のみだった。実データで確認された属性例：
- `annoChar`：注記文字列（日本語。これを描画に使う）
- `knj`：漢字
- `kana`：かな
- `ftCode`：地物コード（前回は一律 100 だった＝種別フィルタに使えない可能性）

**実装前に必ず実タイルを複数ズーム（z11/13/15 など）でデコードして確認すること**：
- `annoChar`（または注記文字列が入る実際のプロパティ名）が何か。`annoChar` でなければ実際のキー名に合わせる。
- そのズーム帯にどれだけの注記が入るか（高ズームほど増える）。
- 注記の種類を区別できる属性があるか（`ftCode` が一律なら、文字列長や `dspPos` 等で重要度を推定するしかない）。

> 前回 `name` 前提で書いて実データと食い違った教訓があるので、ここは**コードを書く前にデコード結果を出力して目視**してから実装に入る。

### wm-sources：basemap.rs に日本語地名取得を追加
```rust
/// 拡大時の地名ラベル（日本語）。
pub struct NameLabelJa {
    pub lat: f64,
    pub lon: f64,
    pub text: String,   // annoChar（日本語注記文字列）
}

impl BaseMapProvider {
    /// label レイヤをデコードし、日本語注記つきの点を返す。
    /// 取得タイルは地図線と同じ z/x/y（現在 app_zoom）。
    /// 高ズーム専用（呼び出しは zoom>=11 のときのみ）。
    pub async fn fetch_labels_ja(&self, bbox: GeoBBox, zoom: u8) -> Result<Vec<NameLabelJa>>;
}
```
- 既存 MVT デコード基盤（fetch_lines と同じ crate）を使い、source-layer を `label` にして各フィーチャの properties から注記文字列を読む。
- 空文字・取得不能はスキップ。

### wm-tui：app.rs / map.rs
```rust
// app.rs
pub name_labels_ja: Option<Vec<wm_sources::basemap::NameLabelJa>>,
```
- zoom が変わったとき、**zoom>=11 なら fetch_labels_ja を非同期取得**（地図取得と同じ経路）、zoom<11 なら None。

map.rs の描画順序（更新）：
```
1. 地図ベース層 lines（海岸→行政界→道路→鉄道）
2. 雨雲（show_radar のときのみ。z>10 はオーバーズームの粗い Grid）
3. 地名ラベル
     - zoom < 11：内蔵英語都市 places::layout_city_labels（従来）
     - zoom >= 11：fetch_labels_ja の日本語地名を画面セルへ投影して描画
4. 中心マーカー ◎
5. タイムライン（show_radar のときのみ）
```

### 日本語描画の注意（全角幅）
- 日本語は端末で **全角＝2 セル幅**。ラベルを置くとき、文字列の表示幅を「全角文字×2 + 半角×1」で計算してレイアウト・右端クリップする。
- 各文字を buf に書く際、全角 1 文字はセル 1 つに char を入れれば ratatui 側が幅 2 として扱う（次セルは空ける）。実装時に ratatui の幅計算（unicode-width 相当）に合わせる。`unicode-width` crate を使って表示幅を出すのが確実。
- 黒背景＋白文字（既存の英語ラベルと同じ可読性処理）。
- 高ズームで地名が増えるので **間引き必須**：画面内の注記が多い場合、同じ行帯で近接するものを先勝ち、または文字列が短い＝より上位の地名を優先（annoChar の長さや種別が使えればそれで）。1 画面あたり最大数（例 30–40）を上限に。

## 実装順序の提案
1. app.rs の zoom 上限 10→16 ＋ radar.rs の RADAR_MAX_ZOOM クランプ。まず「z16 まで地図が精細化し、雨雲は z10 オーバーズームで粗く出る」状態にする（地名はまだ英語のまま）。ここで地図ズームと雨雲ズームの分離を確認。
2. basemap.rs で label レイヤを**実タイル複数ズームでデコードして属性を目視確認**（annoChar の実キー名・件数・種別）。確認結果をコメントに残す。
3. fetch_labels_ja を実装し、wm-tui で zoom>=11 のとき日本語地名を描画。広域=英語／拡大=日本語の切替を確認。
4. 全角幅レイアウト・間引き・◎重なり修正。
5. 受け入れ基準を順に確認。

各段階でコンパイルを通してから次へ。

## 不確実点（前回の教訓を踏まえ明示）
- **最大の不確実点は label レイヤの実属性**。前回 `name`(英語) が無かった。今回使う annoChar（日本語注記）も、実際のプロパティ名・各ズームの件数・種別判別の可否を**コード前にデコード結果で確認**すること。実キー名が `annoChar` でなければ実データに合わせる。
- 雨雲オーバーズームの投影で「取得ズーム(z10)」と「表示ズーム(現在)」を取り違えると雨雲が地図とズレる。z10 でタイル→緯度経度、現在 bbox で緯度経度→セル、を厳密に分けること。早めに z11 で東京駅周辺を出して、雨雲と地図の海岸線が一致するか確認。
- 高ズームでタイル取得枚数は画面サイズ一定ならほぼ一定（高ズームでも増えない）。ただし日本語注記の件数は増えるので描画間引きで対処。
